- **Out-of-Scope Guardrails**:
  - Reject any CLI crate, binary target, `git-outpost`/`gop` executable, help text, exit-code, stdout/stderr formatting, `--no-color`, global `-C`, E2E, or cross-platform whole-binary work. Those are Phase 5.
  - Reject new command flags such as `pull --rebase`, `pull --merge`, `pull --autostash`, `push --source-branch`, or `push --upstream-remote`.
  - Reject broad refactors of existing Phase 0..3 modules unless required to compile the assigned Phase 4 chunk and explicitly justified in that chunk's evidence.
  - Reject unrelated docs cleanup. Product, architecture, and roadmap changes are not expected for these chunks.
  - Reject behavior that silently creates a source branch during `push`; Pu-03 requires `AmbiguousBranchCreation`.
  - Reject `merge`/`rebase` implementations that refresh B from `origin`; users must run `source pull` explicitly.
  - Reject hardcoded `local` for outpost-to-source operations; the configured metadata remote name must drive C-to-B fetch/pull/push.
  - Reject hidden user-visible cross-repo actions; `Reporter::step` must be emitted before each specified source/outpost fetch or push.
  - Reject turning internal safety fetches into mandatory reporter events; the architecture says they are implementation preconditions, not user-visible workflow steps.

- **Review Focus**:
  - P4-C1 reviewers should inspect fast-forward classification carefully: source equal/ahead should not regress, source-behind should fast-forward, true divergence should return `Divergence`, checked-out source branches should update through `git merge --ff-only`, unchecked-out branches should update with `git update-ref`, and pull should surface C/B divergence as typed `Divergence` before the outpost fast-forward.
  - P4-C1 reviewers should also verify `pull` is driven by the attached C branch and metadata remote name, not by CLI assumptions or hardcoded `local`, and that reporter events are emitted before the requested cross-repo actions.
  - P4-C2 reviewers should inspect that wrong-remote and detached-head errors happen before any fetch, that `SourceRemoteRef` branch and remote pieces are validated before use in Git argv, and that merge/rebase do not update B from `origin`.
  - P4-C2 reviewers should check that success tests prove `refs/remotes/<remote>/<branch>` was fetched and that merge/rebase outcomes use simple non-conflicting commits rather than relying on ambiguous Git output.
  - P4-C3 reviewers should inspect side-effect ordering: no push before detached-head, source-missing, source-branch-missing, checked-out-branch-policy, or divergence checks pass. The C-to-B push must use the metadata remote, while B-to-A must use `origin`.
  - P4-C3 reviewers should verify non-bare source behavior: `updateInstead` success is allowed for clean checked-out B, dirty checked-out B returns `GitFailed` with stderr, and `denyCurrentBranch=refuse` returns `PushIntoCheckedOutBranch`.

- **Risks / Cautions**:
  - `SourceRepo::fast_forward_branch_from_origin` is the highest-risk primitive because it must distinguish ref-only updates from checked-out worktree updates without switching B's checkout.
  - Divergence tests can accidentally observe stale refs or raw `GitFailed`; tests should assert the typed `Divergence` paths required by the architecture.
  - Custom remote coverage must prove C-to-B uses metadata while B-to-A remains `origin`; otherwise Phase 4 can pass default-path tests while breaking `--remote-name custom`.
  - Reporter assertions should check `StepKind` ordering and presence, not exact human-facing message text.
  - Push tests that involve checked-out non-bare source branches are sensitive to Git's `receive.denyCurrentBranch` behavior; keep assertions focused on typed errors and required stderr presence.
